# Bug 009: Import targets the source project (from PROJECT_ID in XLS), not the project from which import was initiated

## Summary

When importing a requirements XLS file into project B, SquashTM uses the **`PROJECT_ID` column** in the XLS to determine the target project — it operates on that project's data, **ignoring the context** (project B) from which the import was triggered. As a result, importing an exported file from project A back into project B silently processes project A's requirements. Project B remains empty and no migration occurs.

The export dialog explicitly states the format is *"compatible for an import into SquashTM"*, creating a false expectation that re-importing the exported file will populate the current project with the exported data.

## Environment

- SquashTM Community Edition
- URL: `http://host.docker.internal:8090/squash`
- User: `admin`

## Steps to Reproduce

1. Export requirements from `TestProject` (PROJECT_ID=1) → `Special-Chars Test` folder → XLS file.
2. Create a new empty project `ImportTestProject` (PROJECT_ID=2).
3. Navigate to `ImportTestProject` → Requirements workspace.
4. Trigger import: toolbar → import/export → Import.
5. Upload the XLS file exported in step 1 and confirm.
6. Examine the import report and the `ImportTestProject` requirements tree.

## Expected Result

- New requirements are created inside `ImportTestProject` with new IDs (requirement-folder ID, requirement IDs, etc. all belonging to project 2).
- `ImportTestProject` contains a new folder `Special-Chars Test` with SC-001 through SC-008 (9 rows → versions handled correctly).
- The import report reads: "9 rows created in ImportTestProject".

## Actual Result

- The import processes requirements in `TestProject` (PROJECT_ID=1, as specified in the XLS), NOT in `ImportTestProject`.
- `ImportTestProject` remains empty of its own requirements.
- Clicking on `SC-001` in the combined tree view navigates to `/requirement/5` (TestProject's original requirement, REQ_ID=5).
- The `Special-Chars Test` folder in the tree resolves to `/requirement-folder/4` (TestProject's original folder ID=4).
- The import report shows "7 successful, 2 warnings" — these are hits against the already-existing TestProject requirements, not new records.

## Evidence

| Element | URL after click | Belongs to |
|---------|-----------------|------------|
| SC-001 (clicked from ImportTestProject view) | `/requirement/5` | TestProject |
| Special-Chars Test folder | `/requirement-folder/4` | TestProject |

## Root Cause

The XLS export embeds `PROJECT_ID=1` (the source project). When the importer reads this column, it routes all processing to the project with that ID, regardless of which project's import dialog was used.

## Impact

- **Migration is impossible**: users cannot use the export/import cycle to migrate requirements from one project to another without manually editing every `PROJECT_ID` cell in the XLS.
- **Silent data loss**: users believe they have imported requirements into a new project, but the new project is empty. The import report says "successful" because it silently operated on the source project's data.
- **False "compatible for import" claim**: the export dialog's guarantee of import compatibility is broken for cross-project scenarios (the most common use case for import).
- Related to **Bug 004**: the missing `ACTION` column further prevents any meaningful import behavior, since without `ACTION=C` ("Create"), the importer defaults to updating existing records rather than creating new ones.
