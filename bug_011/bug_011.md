# Bug 011: Import from non-rich-text export corrupts descriptions containing angle-bracket sequences

## Summary

When importing a requirements XLS that was exported **without** "Keep the rich text format", any description text containing `<...>` sequences (e.g. `<special>`) is silently corrupted: the `<...>` content is stripped as if it were an HTML tag. This causes permanent data loss for requirements whose descriptions contain angle-bracket characters.

## Environment

- SquashTM Community Edition
- URL: `http://host.docker.internal:8090/squash`
- User: `admin`

## Steps to Reproduce

1. Create a requirement with a description containing an angle-bracket sequence:
   - Description (rich text editor): `This req has <special> & "quoted" chars`
2. Export the folder to XLS with **"Keep the rich text format" unchecked**.
3. Confirm the exported `REQ_VERSION_DESCRIPTION` cell value is: `This req has <special> & "quoted" chars` (plain text, no HTML encoding).
4. Import the exported XLS file back into SquashTM.
5. Navigate to the requirement and inspect its description.

## Expected Result

The description is unchanged:

```
This req has <special> & "quoted" chars
```

(Either the importer correctly HTML-escapes `<special>` to `&lt;special&gt;` before storing, or it stores the raw text and displays it as-is.)

## Actual Result

The `<special>` sequence is silently stripped. The description becomes:

```
This req has & "quoted" chars
```

The `<special>` text is permanently lost. Both Version 1 and Version 2 of the requirement are affected when they contain the same description.

## Root Cause Analysis

The non-rich-text export stores description content as plain text without HTML-encoding `<` and `>`. When the importer stores this value in SquashTM's rich-text description field (which expects HTML), the stored value contains a raw `<special>` string. SquashTM's rich-text renderer then parses `<special>` as an unknown HTML tag and strips it on display — or the importer itself strips it during HTML sanitization before saving.

The importer should HTML-encode angle brackets from non-rich-text imports before storing them in the rich-text field.

## Impact

- **Permanent data corruption**: any requirement description containing `<text>` patterns is silently corrupted upon import from non-rich-text export. The original data is unrecoverable from within SquashTM.
- **Data loss in the source project**: because Bug 009 causes the import to operate on the source project rather than the target project, the original TestProject requirements are corrupted — not just a hypothetical import target.
- **Silent failure**: the import log shows "Successful" for these rows, with no warning about the content change.

## Workaround

Use the **"Keep the rich text format"** export option to preserve HTML encoding of the description. The rich text export correctly handles all versions and TC links, making it the safer choice for round-trip import — though importing with `PROJECT_ID` from a different project still causes data to be placed in the source project (see **Bug 009**).

## Notes

- The `&` character in `& "quoted" chars` is correctly preserved (not double-encoded) in both export modes.
- The `"` (double-quote) characters are also correctly preserved.
- Only `<...>` sequences are affected — they are stripped as if they were HTML tags.
