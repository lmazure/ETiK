# Bug 002 — Forward slash in requirement name creates ambiguous `REQ_PATH`

## Summary

When a requirement name contains a forward slash (`/`), the `REQ_PATH` in the export is ambiguous and cannot be reliably parsed back. The path last segment no longer matches the requirement's name.

## Steps to Reproduce

1. Log in as `admin`.
2. Create a requirement with a forward slash in its name, e.g. `REQ-SC06 Path/\\Test`.
3. Select the parent folder and choose **Import/Export → Export**.
4. Open the downloaded `.xls` file and inspect the `REQ_PATH` and `REQ_VERSION_NAME` columns.

## Expected Result

The path should either:
- **Option A**: reject `/` in requirement names with a validation error, or
- **Option B**: use a documented, unambiguous escaping convention consistently.

For option B, the documented convention should be stated in the import/export specification, and an import of the exported file should round-trip correctly.

## Actual Result

| Column | Value |
|--------|-------|
| `REQ_VERSION_NAME` | `REQ-SC06 Path/\\Test` |
| `REQ_PATH` | `/TestProject/Special-Chars Test/REQ-SC06 Path\/\\Test` |

The `/` in the name is silently escaped to `\/` in the path **without documentation**. The last path segment (`\\Test`) does not match the requirement name (`REQ-SC06 Path/\\Test`).

A parser that splits `REQ_PATH` on `/` would produce the wrong segments:
```
['', 'TestProject', 'Special-Chars Test', 'REQ-SC06 Path', '\\Test']
```
Instead of the correct:
```
['', 'TestProject', 'Special-Chars Test', 'REQ-SC06 Path/\\Test']
```

## Impact

- **Import round-trip broken** — importing the exported file would either fail or create the requirement in the wrong location with the wrong name.
- **Undocumented escaping** — the `\/` escape convention is not mentioned anywhere in the import/export documentation.

## Environment

- SquashTM instance: `http://host.docker.internal:8090/squash`
- Project: `TestProject`, folder: `Special-Chars Test`
- Requirement: `REQ-SC06 Path/\\Test`
